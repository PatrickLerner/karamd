# karamd recurring rules (`.taskmd.recurring.yaml`)

karamd's `generate` command reads a rules file, inspects the existing tasks in a
vault, and creates a new task file **only when a rule is due**. It reads task
*state* each run rather than emitting on a fixed schedule, so re-running the
same day never duplicates (idempotent). The rules file default location is
`<vault>/.taskmd.recurring.yaml`, kept next to `.taskmd.yaml`.

The file is a **YAML list** of rules. Each rule materialises taskmd-compatible
task files when due (see the `taskmd-format` skill for the file shape).

## Rule fields

Every rule requires:

| Field | Meaning |
|-------|---------|
| `key` | Dedup marker, unique across the whole file. Groups a rule's tasks. |
| `title` | Task title (becomes the `# <title>` heading). |
| `trigger` | One of `after_completion`, `calendar`, `monthly`, `weekly`, `nth_weekday`. |

Trigger-specific fields:

- **`after_completion`**: `every_days` (int). The next task is due
  `every_days` days after the *last occurrence's completion*. Interval-from-
  completion, not a fixed calendar. Self-healing: a missed run just catches up.
- **`calendar`**: `annual` (`"MM-DD"`) + `lead_days` (int). Due `lead_days`
  before a fixed annual date, once per year. `"02-29"` clamps to `02-28` in
  common years so the rule still fires yearly.
- **`monthly`**: `day_of_month` (1-31) + `lead_days` (0-27). Due `lead_days`
  before that day of the month, once per month. Days 29-31 clamp to the month's
  last day, so `31` still fires in February. `lead_days` is capped at 27 so
  windows never overlap.
- **`weekly`**: `day_of_week` (`mon`|`tue`|`wed`|`thu`|`fri`|`sat`|`sun`, exactly
  that lowercase three-letter form; anything else is rejected). Due on that
  weekday, once per ISO week. A run on or after the day within the same ISO week
  still fires (Sat/Sun catch up a missed Friday); a fully missed week is not
  backfilled. Like `after_completion`, an open task for the key blocks a second,
  so there is never more than one at a time. No `lead_days`: weekly is strictly
  on-or-after the day.
- **`nth_weekday`**: `day_of_week` (same form as weekly) + `week` (`1`-`4` or
  `last`). Due on the Nth (or last) matching weekday of the month, once per
  month, e.g. "first Monday", "last Friday". On-or-after that date so a late run
  catches up within the month; an open task for the key blocks a second.

`weekly`, `monthly`, and `nth_weekday` also accept an optional **`interval`**
(integer >= 1, default 1 = every period) and **`anchor`** (`YYYY-MM-DD`, a date
on the desired cadence). `interval: 2` means every other period (biweekly, every
other month). `interval: 1` or omitted is a no-op. Omitting `anchor` aligns the
cadence to a fixed epoch (still deterministic). The dedup marker is unchanged;
`interval` only gates which periods are eligible.

Optional on any rule:

| Field | Meaning |
|-------|---------|
| `phase` | Sets the task's `phase`. |
| `priority` | `low`, `medium`, `high`, `critical`. |
| `tags` | YAML list of tag strings. |
| `body` | Markdown replacing the default `## Objective / ## Tasks / ## Acceptance Criteria` TODO stub. |

### The optional `body`

When present, `body` replaces the default TODO stub verbatim; when absent the
stub is emitted unchanged. karamd **always** writes the frontmatter, the
`# <title>` heading, and a `<!-- Generated by karamd ... -->` provenance comment
regardless of `body`: the body is only what comes after the heading. Use a `|`
block scalar. An empty or whitespace-only `body` is rejected (it is worse than
the stub). Do not start a body with a `---` fence: taskmd/Obsidian would read it
as a second frontmatter block.

## Dedup markers karamd stamps on generated tasks

karamd writes a `recurring:` frontmatter field on every generated task; this is
how it avoids duplicates. The exact form encodes the period:

- `after_completion` → `recurring: <key>`
- `calendar` → `recurring: "<key>:<year>"`
- `monthly` → `recurring: "<key>:YYYY-MM"`
- `weekly` → `recurring: "<key>:YYYY-Www"` (ISO week, e.g. `linkedin:2026-W28`;
  the ISO year can differ from the calendar year near Jan 1)
- `nth_weekday` → `recurring: "<key>:YYYY-MM"` (same month discriminator as
  `monthly`)

The period suffix is what makes "once per period" hold even if the task is
completed early inside its lead window. This is an unknown field to taskmd, so
taskmd/Obsidian preserve it verbatim on edit (see `taskmd-format`).

### How due-ness reads state (after_completion)

An *open* task for the key blocks re-creation. Otherwise the rule is due when
`today - last_occurrence >= every_days`, where `last_occurrence` is the most
recent terminal task's `completed_at`, else `cancelled_at`, else `created_at`.
Cancelling an occurrence keeps the series: the next one schedules `every_days`
after the `cancelled_at`, not immediately. A never-run key is due.

### How due-ness reads state (weekly)

Two guards, both applied. First, an *open* task for the key blocks a second (so a
lingering task from a prior week does not pile up). Then, if today's ISO weekday
is on or after `day_of_week`, the current ISO week's `YYYY-Www` marker is used;
if a task already carries that marker (e.g. it was completed early this week),
nothing is created. The discriminator is always the *current* ISO week, so a
fully skipped week is never backfilled.

## Example file

```yaml
# Interval-from-completion: next task ~18 days after finishing the previous one.
- key: periodic-checkin
  title: "Reach out to [[Someone]]"
  trigger: after_completion
  every_days: 18
  phase: next
  priority: medium
  tags: [personal]

# Fixed annual date, once per year, surfaced 10 days early.
- key: annual-birthday
  title: "Birthday: present & congratulations"
  trigger: calendar
  annual: "07-20"
  lead_days: 10
  priority: high
  tags: [personal]

# Fixed day each month, once per month (marker key:YYYY-MM), 7 days early.
- key: monthly-topup
  title: "Top up account"
  trigger: monthly
  day_of_month: 12
  lead_days: 7
  priority: medium
  tags: [finance]

# Fixed weekday, once per ISO week (marker key:YYYY-Www). Lands on Friday;
# a later run in the same week catches up.
- key: linkedin-weekly
  title: "Evaluate and schedule LinkedIn posts"
  trigger: weekly
  day_of_week: fri
  priority: medium
  tags: [marketing]

# Nth (or last) weekday of the month, once per month (marker key:YYYY-MM).
- key: ops-review
  title: "Monthly ops review"
  trigger: nth_weekday
  day_of_week: mon
  week: 1            # first Monday; `last` for the last such weekday
  priority: medium
  tags: [ops]

# Every other ISO week (biweekly), aligned to a Friday on the cadence.
- key: sprint-retro
  title: "Sprint retro"
  trigger: weekly
  day_of_week: fri
  interval: 2
  anchor: 2026-07-10
  priority: medium
  tags: [team]

# Optional body replacing the TODO stub.
- key: quarterly-backup
  title: "Verify backups restore cleanly"
  trigger: after_completion
  every_days: 90
  priority: high
  tags: [ops]
  body: |
    ## Objective

    Confirm the offsite backup actually restores, not just that it runs.

    ## Tasks

    - [ ] Pull the latest snapshot
    - [ ] Restore to a scratch host
    - [ ] Spot-check a known record

    ## Acceptance Criteria

    - A restored copy opens and the spot-checked record matches
```

## Common mistakes

- Non-unique `key`s: they must be unique across the whole file.
- Using `done`/`in_progress` anywhere: statuses are `completed`, `in-progress`,
  `in-review` (see `taskmd-format`).
- `lead_days` above 27 on `monthly`: rejected.
- A field owned by another trigger (e.g. `day_of_month` on a `weekly` rule, or
  `day_of_week` on `monthly`): rejected. Each trigger accepts only its own
  fields, so a misplaced field fails loudly instead of being ignored.
- Assuming karamd emits on a schedule: it reads task state each run. Editing or
  completing tasks in taskmd/Obsidian is fully accounted for.
